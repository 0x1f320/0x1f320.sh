## 왜 블로그를 만들었는가

메모장에 적어두는 것과 글로 정리하는 것은 다릅니다. 글을 쓰면 **생각이 정리**되고, 정리된 생각은 나중에 쓸모가 있습니다. 그게 *나한테든* 다른 사람한테든.

사실 블로그를 만들겠다고 마음먹은 건 꽤 오래 전인데, 매번 "좋은 글감이 생기면 시작해야지" 하고 미뤘습니다. 그래서 이번엔 ~~완벽한 준비를 하고~~ 그냥 만들었습니다.

## 기술 스택

이 블로그는 개인 사이트 안에 포함되어 있고, 스택은 다음과 같습니다.

| 영역 | 기술 | 비고 |
|------|------|------|
| 프레임워크 | Next.js 16 | App Router, Turbopack |
| 콘텐츠 | MDX | 마크다운 + React |
| 다국어 | next-intl | ko / en |
| 댓글 | giscus | GitHub Discussions |
| 배포 | Vercel | 정적 생성 |

### 글 작성 방식

글은 `content/blog/` 디렉토리에 `.mdx` 파일로 작성합니다. 파일명 규칙:

```
content/blog/
├── hello-world.ko.mdx
└── hello-world.en.mdx
```

각 파일의 frontmatter에 메타데이터를 넣습니다:

```yaml
---
title: "글 제목"
date: "2026-04-03"
description: "한 줄 설명"
---
```

### 코드 하이라이팅

TypeScript 예시:

```typescript
interface BlogPost {
  slug: string;
  title: string;
  date: string;
  content: string;
}

async function getBlogPost(slug: string, locale: string): Promise<BlogPost | null> {
  const filePath = path.join(CONTENT_DIR, `${slug}.${locale}.mdx`);
  const raw = await fs.readFile(filePath, "utf-8");
  const { data, content } = matter(raw);

  return {
    slug,
    title: data.title ?? slug,
    date: data.date ?? "",
    content,
  };
}
```

인라인 코드도 있습니다. `pnpm dev`로 개발 서버를 실행하고, `pnpm build`로 빌드합니다.

## 블로그를 만들며 배운 것

1. **MDX는 생각보다 간단합니다.** `next-mdx-remote`를 쓰면 파일시스템에서 읽어서 서버 컴포넌트에서 바로 렌더할 수 있습니다.
2. **정적 생성이 빠릅니다.** `generateStaticParams`로 모든 글을 빌드 타임에 생성하면 배포 후 로딩이 거의 즉시입니다.
3. **다국어 지원은 초반에 넣어야 합니다.** 나중에 넣으면 모든 텍스트를 다시 만져야 합니다.

그리고 몇 가지 하지 말아야 할 것도 배웠습니다:

- 완벽한 디자인을 기다리지 마세요
- 첫 글부터 대단한 내용을 쓰려 하지 마세요
- CMS를 도입하려다 시간만 쓰지 마세요 — 파일 시스템으로 충분합니다

> 완벽한 첫 글을 쓰려고 하면 영원히 시작하지 못합니다.
